# 重试/修复机制 (Retry/Fix Mechanism)

## 概述

重试/修复机制是一个自动质量改进系统，当agent的回答评估分数过低时，会自动触发重试以提升回答质量。

**实现日期**: 2026-05-23

## 核心功能

### 自动触发条件

当满足以下任一条件时触发重试：
- **总体评分 < 6.0** (低于"良好"水平)
- **准确性评分 < 5.0** (严重质量问题)

### 两种策略

#### 1. FIX 策略（修复）

**使用场景**：
- 来源正确但叙述有问题
- 连贯性评分低 (< 6.0)
- 合理性评分低 (< 6.0)
- 准确性和相关性可接受 (≥ 5.0)

**工作方式**：
- 保留已搜索的新闻来源
- 在对话中追加改进提示
- 要求LLM改进叙述结构和推理
- 不重新搜索新闻

**成本**：约1.3x token消耗

#### 2. REDO 策略（重做）

**使用场景**：
- 准确性 < 5.0（来源错误或幻觉）
- 相关性 < 4.0（答非所问）
- 没有找到任何来源
- 需要完全重新开始

**工作方式**：
- 重置对话历史（保留系统消息）
- 重新搜索新闻
- 从头开始回答问题
- 使用改进的搜索策略

**成本**：约2.0x token消耗

### 智能策略选择

系统根据评估结果自动选择最合适的策略：

```
评估分数
    ↓
准确性 < 5.0? ────→ YES ──→ REDO
    ↓ NO
相关性 < 4.0? ────→ YES ──→ REDO
    ↓ NO
无来源? ──────────→ YES ──→ REDO
    ↓ NO
总分 < 6.0? ──────→ YES ──→ FIX
    ↓ NO
不需要重试
```

## 架构设计

### 文件结构

```
financial_news_agent/
├── retry_manager.py      # 核心模块
│   ├── RetryConfig       # 配置类
│   ├── decide_retry_strategy()  # 策略决策
│   ├── build_fix_prompt()       # FIX提示词生成
│   └── build_redo_prompt()      # REDO提示词生成
│
├── agent.py
│   ├── run_agent()              # 原有agent循环（不变）
│   └── run_agent_with_retry()   # 新增包装器函数
│
└── __main__.py
    └── 使用 run_agent_with_retry() 替代 run_agent()
```

### 执行流程

```
用户查询
    ↓
run_agent_with_retry() 开始
    ↓
┌─────────────────────────┐
│ 尝试 1: run_agent()     │
│ - 搜索新闻              │
│ - 生成回答              │
│ - 自我评估              │
└─────────────────────────┘
    ↓
评估分数是否合格?
    ↓ NO (分数 < 阈值)
决定策略 (FIX/REDO)
    ↓
打印重试信息给用户
    ↓
修改对话历史
    ├─ FIX: 追加改进提示
    └─ REDO: 重置+重新搜索提示
    ↓
┌─────────────────────────┐
│ 尝试 2: run_agent()     │
│ - 根据策略执行          │
│ - 生成改进的回答        │
│ - 再次评估              │
└─────────────────────────┘
    ↓
返回最终结果
    ├─ 回答
    ├─ 评估分数
    ├─ 来源列表
    └─ 重试历史 (如果启用显示)
```

## 配置选项

### 环境变量

在 `.env` 文件中配置：

```bash
# 启用/禁用
RETRY_ENABLE=true                    # true|false

# 阈值设置
RETRY_THRESHOLD_OVERALL=6.0          # 总分阈值 (1.0-10.0)
RETRY_THRESHOLD_ACCURACY=5.0         # 准确性阈值 (1.0-10.0)

# 重试限制
RETRY_MAX_ATTEMPTS=1                 # 最大重试次数 (0-3)

# 策略控制
RETRY_STRATEGY=auto                  # auto|fix|redo|disabled
                                     # auto: 智能选择
                                     # fix: 总是使用FIX
                                     # redo: 总是使用REDO
                                     # disabled: 禁用重试

# 用户界面
RETRY_SHOW_ATTEMPTS=true             # 是否显示重试历史
```

### 推荐配置

**生产环境**（平衡质量和成本）：
```bash
RETRY_ENABLE=true
RETRY_THRESHOLD_OVERALL=6.0
RETRY_THRESHOLD_ACCURACY=5.0
RETRY_MAX_ATTEMPTS=1
RETRY_STRATEGY=auto
RETRY_SHOW_ATTEMPTS=false
```

**开发环境**（更高质量要求）：
```bash
RETRY_ENABLE=true
RETRY_THRESHOLD_OVERALL=7.0
RETRY_THRESHOLD_ACCURACY=6.0
RETRY_MAX_ATTEMPTS=2
RETRY_STRATEGY=auto
RETRY_SHOW_ATTEMPTS=true
```

**测试环境**（禁用重试）：
```bash
RETRY_ENABLE=false
```

## 成本分析

### Token消耗

| 场景 | 策略 | Token倍数 | 示例 (基准5000) |
|------|------|-----------|----------------|
| 无需重试 | - | 1.0x | 5,000 tokens |
| FIX一次 | Fix | 1.3x | 6,500 tokens |
| REDO一次 | Redo | 2.0x | 10,000 tokens |

### 成本控制措施

1. **最大重试限制**：默认1次，最多2次总尝试
2. **智能策略选择**：优先使用成本更低的FIX
3. **早期退出**：一旦达到质量标准立即停止
4. **可配置开关**：可完全禁用重试功能

### 预期影响

- **平均成本增加**：约15-20%（大部分查询无需重试）
- **最坏情况**：2.0x（REDO一次）
- **质量提升**：显著减少低质量回答

## 边界情况处理

| 情况 | 处理方式 |
|------|----------|
| 评估失败 | 使用默认分数(3/10)，不触发重试 |
| 达到最大重试次数 | 返回最后一次尝试，打印警告 |
| 重试时API错误 | 捕获异常，返回上一次结果 |
| Token超限 | 现有context_manager处理 |
| 重做后仍无来源 | 算作失败，不再重试 |
| 重试后分数更低 | 仍然返回（已尽力改进） |
